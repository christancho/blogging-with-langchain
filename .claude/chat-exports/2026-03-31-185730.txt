⏺ Here's the situation:

**Root cause:** `Config.get_llm()` uses LangChain's `with_fallbacks()`, but that
  only triggers on a specific set of exceptions by default. A 529 overload error
  from Anthropic isn't in that list — so it propagates up, hits the bare `except
  Exception as e` in `writer_node`, returns empty content, and the run dies.

**The OpenRouter fallback you have configured isn't actually firing for 529s.**

---

Two decisions to make:

**1. Retry vs. fallback immediately**

A 529 is transient — Anthropic is just busy. Jumping straight to OpenRouter
  burns cost and changes model behavior mid-run. A better first response is
  retry with exponential backoff (e.g. 3 attempts: 2s, 4s, 8s). Only fall back
  to OpenRouter if retries are exhausted.

**2. Where to fix it**

- **Option A — Fix in `Config.get_llm()`**: Add `.with_retry()` on the Anthropic
  LLM before `.with_fallbacks()`. Clean, applies everywhere automatically.
- **Option B — Fix at the call site in each node**: Wrap `chain.invoke()` with a
  retry loop. More granular but duplicates logic across nodes.

Option A is the right call — one place to change, all nodes benefit.

---

**Proposed implementation in `Config.get_llm()`:**

```python
anthropic_llm = ChatAnthropic(...)
anthropic_with_retry = anthropic_llm.with_retry(
    stop_after_attempt=3,
    wait_exponential_jitter=True
)
return anthropic_with_retry.with_fallbacks([openrouter_llm])
```

This retries Anthropic up to 3 times on any error (including 529), then falls
  back to OpenRouter if all retries fail. Want me to implement this?
